ABSTRAK

Implementasi Factory Pattern & Chain of Responsibility pada Workflow Persetujuan Pengadaan Barang/Jasa. Penelitian ini bertujuan membangun sistem approval pengadaan berjenjang yang adaptif terhadap nilai transaksi menggunakan pendekatan design pattern. Berdasarkan studi awal pada 3 perusahaan skala menengah, ditemukan keluhan utama berupa ketidakjelasan jenjang approval (100% responden) dan minimnya riwayat persetujuan (100%). Sistem yang diusulkan menerapkan Factory Pattern guna membuat workflow handler secara terpusat serta Chain of Responsibility untuk memproses persetujuan sesuai nominal (≤50 jt, 51–250 jt, >250 jt).

Implementasi menggunakan Spring Boot (Java 11) dengan tiga role: Manager, Kepala Divisi, dan Direksi. Setiap approval mencatat riwayat lengkap (role, remark, timestamp). REST API disediakan untuk antarmuka web (HTML/CSS/JS). Hasil user testing terhadap 3 responden menunjukkan tingkat keberhasilan 88.9% setelah iterasi perbaikan pada tombol approve dan tampilan status. Sistem telah dideploy di AWS EC2 (t2.micro) dan berjalan stabil. Penelitian ini membuktikan bahwa kombinasi Factory Pattern dan Chain of Responsibility menghasilkan workflow approval yang terstruktur, fleksibel, dan mudah dipelihara.

Kata Kunci: Factory Pattern, Chain of Responsibility, workflow approval, pengadaan berjenjang, audit trail, Spring Boot, AWS EC2.

BAB 1 – PENDAHULUAN

1.1 Latar Belakang

Dalam rekayasa perangkat lunak, design pattern memberikan solusi umum untuk masalah yang berulang. Untuk sistem approval yang aturannya bergantung pada nilai nominal, pendekatan prosedural akan sulit dipelihara. Factory Pattern dan Chain of Responsibility dipilih karena mampu memisahkan pembuatan workflow dari logika bisnis, serta memberikan fleksibilitas jika aturan berubah di masa depan.

Berdasarkan observasi awal terhadap 3 perusahaan skala menengah, ditemukan bahwa 100% responden internal mengeluhkan ketidakjelasan jenjang approval dan kurangnya dokumentasi riwayat persetujuan. Oleh karena itu, dibangun sistem approval berjenjang dengan ketentuan:

  • ≤ 50 juta : cukup disetujui oleh Manager → status APPROVED.
  • 51 – 250 juta : Manager → Kepala Divisi → APPROVED.
  • > 250 juta (maksimal 1 Milyar) : Manager → Kepala Divisi → Direksi → APPROVED.

Sistem mencatat history approval (role, remark, timestamp) sebagai audit trail. Antarmuka web disediakan agar ketiga role dapat mensimulasikan approval secara manual.

1.2 Tujuan Penelitian

  • Menerapkan Factory Pattern untuk membuat workflow approval secara terpusat dan fleksibel.
  • Mengimplementasikan Chain of Responsibility agar setiap level approval dapat diproses secara manual tanpa auto-forward.
  • Menyediakan REST API dan antarmuka web (HTML/CSS/JS) untuk simulasi tiga role.
  • Mendokumentasikan seluruh tahap desain, kode, dan deployment di AWS EC2.

1.3 Objek Riset

  • Aplikasi: Sistem Approval Pengadaan Berjenjang (Java Spring Boot)
  • Fitur yang dianalisis: workflow approval eksisting (belum terstruktur)
  • Fitur yang diusulkan: workflow engine berbasis Factory Pattern + Chain of Responsibility

BAB 2 – METODOLOGI & DESAIN PATTERN

2.1 Design Pattern yang Digunakan

Pattern Lokasi Implementasi Manfaat
Factory Pattern ApprovalHandlerFactory.createWorkflow() Memusatkan pembuatan rantai handler (Manager → DivHead → Director), memudahkan perubahan urutan/level tanpa mengubah service. Menerapkan prinsip Open/Closed Principle.
Chain of Responsibility ManagerHandler, DivisionHeadHandler, DirectorHandler Setiap handler hanya memproses jika status sesuai; setelah itu berhenti (tidak auto-forward). Memberikan kontrol manual per role dan memungkinkan penambahan level approval baru tanpa mengubah handler lain.
MVC Pattern ProcurementRequest (Model), ApprovalController (Controller), index.html (View) Memisahkan logika bisnis (Model), kontrol (Controller), dan tampilan (View). Memudahkan pengembangan, testing, dan pemeliharaan kode.

BAB 3 – PENJELASAN FACTORY PATTERN & CHAIN OF RESPONSIBILITY

3.1 Pengertian Factory Pattern

Factory Pattern adalah design pattern creational yang menyediakan antarmuka untuk membuat objek tanpa mengekspos logika pembuatan objek kepada klien. Dalam konteks sistem approval berjenjang, Factory Pattern digunakan untuk membuat dan merangkai handler-handler approval (Manager, Kepala Divisi, Direksi) menjadi sebuah workflow engine yang siap pakai.

📌 Bagaimana Factory Pattern Bekerja pada Sistem Ini:
  1. Kelas ApprovalHandlerFactory memiliki method statis createWorkflow().
  2. Method ini membuat instance dari ManagerHandler, DivisionHeadHandler, dan DirectorHandler.
  3. Ketiga handler tersebut dirangkai berurutan menggunakan method setNext(): Manager → DivisionHead → Director.
  4. Handler pertama (Manager) dikembalikan ke pemanggil (ApprovalService).
  5. Dengan demikian, seluruh logika perangkaian handler disembunyikan di dalam Factory, dan service hanya perlu memanggil createWorkflow() tanpa tahu detail perangkaiannya.
// Factory Pattern: membuat rantai handler tanpa mengekspos logika perangkaian public class ApprovalHandlerFactory { public static ApprovalHandler createWorkflow() { ManagerHandler manager = new ManagerHandler(); // Buat handler Manager DivisionHeadHandler division = new DivisionHeadHandler(); // Buat handler Kepala Divisi DirectorHandler director = new DirectorHandler(); // Buat handler Direksi manager.setNext(division); // Rangkai: Manager → DivisionHead division.setNext(director); // Rangkai: DivisionHead → Director return manager; // Kembalikan handler pertama sebagai entry point } }

3.2 Pengertian Chain of Responsibility

Chain of Responsibility adalah design pattern behavioral yang memungkinkan suatu request melewati rantai handler, di mana setiap handler memiliki kesempatan untuk memproses request atau meneruskannya ke handler berikutnya. Dalam sistem approval berjenjang, setiap level approval (Manager, Kepala Divisi, Direksi) adalah sebuah handler dalam rantai.

📌 Bagaimana Chain of Responsibility Bekerja pada Sistem Ini:
  1. Setiap handler mengimplementasikan interface ApprovalHandler dengan method handle().
  2. Ketika method handle() dipanggil, handler akan memeriksa apakah status request sesuai dengan tanggung jawabnya.
  3. Jika sesuai, handler akan:
    • Mencatat history approval
    • Memperbarui status request menggunakan aturan bisnis (nextStatusAfter)
    • TIDAK otomatis melanjutkan ke handler berikutnya - ini memastikan approval manual per level
  4. Jika status tidak sesuai, handler akan meneruskan request ke handler berikutnya dalam rantai (jika ada).
  5. Dengan mekanisme ini, alur approval berjenjang dapat berjalan: Manager (level 1) → Kepala Divisi (level 2) → Direksi (level 3).
// Contoh implementasi Chain of Responsibility pada ManagerHandler public class ManagerHandler implements ApprovalHandler { private ApprovalHandler next; // Referensi ke handler berikutnya (Kepala Divisi) @Override public void handle(ProcurementRequest request) { if (request.getStatus() == ApprovalStatus.WAITING_MANAGER) { // 1. Proses approval oleh Manager request.addHistory("MANAGER", "Manager menyetujui..."); request.setStatus(request.nextStatusAfter(ApprovalStatus.WAITING_MANAGER)); // 2. TIDAK LANJUT OTOMATIS - menunggu user role berikutnya klik approve } else if (next != null) { next.handle(request); // Lanjut ke handler berikutnya jika status tidak sesuai } } }

3.3 Alur Approval Berjenjang dari Manager sampai Direktur

📊 Alur Lengkap Approval Berjenjang:

1. Pengajuan Baru → Status: WAITING_MANAGER
↓
2. Manager Approve → System > Handler Manager memproses → Update status berdasarkan nominal:
   - ≤ 50jt → APPROVED (selesai)
   - > 50jt → WAITING_DIVISION_HEAD
↓
3. Kepala Divisi Approve (jika status WAITING_DIVISION_HEAD) → Update status:
   - ≤ 250jt → APPROVED (selesai)
   - > 250jt → WAITING_DIRECTOR
↓
4. Direksi Approve (jika status WAITING_DIRECTOR) → Update status menjadi APPROVED (selesai)
↓
5. Selesai → Riwayat approval terekam lengkap dengan timestamp

3.4 Keuntungan Penggunaan Kedua Pattern

Keuntungan Penjelasan
FleksibilitasMudah menambah atau mengubah urutan level approval (misal menambah level VP) tanpa mengubah kode yang sudah ada.
Pemisahan Tanggung JawabSetiap handler hanya fokus pada satu level approval, sesuai prinsip Single Responsibility.
ReusabilityFactory dapat digunakan kembali untuk membuat workflow dengan konfigurasi berbeda.
Open/Closed PrincipleKode terbuka untuk ekstensi (menambah level baru) tetapi tertutup untuk modifikasi (tidak mengubah handler lama).
Manual ControlTidak ada auto-forward, setiap level harus klik approve secara manual - sesuai kebutuhan bisnis riil.

BAB 5 – PENJELASAN METODE PROGRAM JAVA

Setiap class memiliki tanggung jawab spesifik dalam sistem approval berjenjang ini:

  • ProcurementRequest.java: Model data yang menyimpan informasi pengajuan dan riwayat approval. Method nextStatusAfter() mengimplementasikan aturan bisnis penentuan status berikutnya berdasarkan nominal.
  • ApprovalStatus.java: Enum yang mendefinisikan 4 status (WAITING_MANAGER, WAITING_DIVISION_HEAD, WAITING_DIRECTOR, APPROVED).
  • ApprovalHandler (Interface): Kontrak untuk semua handler dalam Chain of Responsibility dengan method handle() dan setNext().
  • ManagerHandler, DivisionHeadHandler, DirectorHandler: Implementasi konkret Chain of Responsibility. Setiap handler hanya memproses jika status sesuai, lalu memperbarui status. Tidak ada auto-forward untuk memastikan approval manual per level.
  • ApprovalHandlerFactory: Implementasi Factory Pattern yang membuat dan merangkai ketiga handler menjadi workflow engine.
  • ApprovalService: Service layer yang menyimpan data (Map in-memory) dan mengatur logika bisnis seperti submit request, filter pending request, dan menjalankan workflow saat approve.
  • ApprovalController: REST Controller yang menyediakan endpoint untuk frontend (GET/POST).
  • ProcurementApplication: Main class Spring Boot dengan annotation @SpringBootApplication.

BAB 8 – DEPLOYMENT & OPERASI (AWS EC2)

8.1 Persiapan EC2 Instance

📌 Fungsi: EC2 (Elastic Compute Cloud) adalah layanan AWS untuk menyewa server virtual. Instance t2.micro dipilih karena termasuk dalam free tier. Security Group berfungsi sebagai firewall virtual untuk mengontrol traffic masuk.
  • Buat instance EC2 t2.micro (Amazon Linux 2023 atau Ubuntu 22.04)
  • Pada Security Group, buka port 8081 (aplikasi) dan 22 (SSH)
  • Catat Public IP instance

8.2 Instal Java 11 di EC2

📌 Fungsi: Aplikasi Spring Boot membutuhkan Java Runtime Environment (JRE) versi 11 untuk menjalankan file JAR. Perintah di bawah menginstal OpenJDK 11, yaitu implementasi open-source dari Java SE.
# Untuk Ubuntu/Debian sudo apt update && sudo apt install openjdk-11-jdk -y # Update package dan install JDK 11 # Untuk Amazon Linux sudo dnf install java-11-amazon-corretto-devel # Install Corretto JDK 11 # Verifikasi instalasi java -version # Output harus menampilkan versi 11.x.x

8.3 Build & Jalankan Aplikasi

📌 Fungsi: Maven digunakan untuk mengcompile dan mengemas aplikasi menjadi file JAR. Perintah nohup dan & digunakan untuk menjalankan aplikasi di background agar tetap berjalan meskipun sesi SSH ditutup.
# Build dengan Maven (hasilkan file JAR) mvn clean package # Membersihkan dan mengcompile ulang proyek # Jalankan foreground (untuk testing langsung) java -jar target/procurement-approval-factory-1.0.jar --server.port=8081 # Jalankan di port 8081 # Jalankan background (tahan close terminal) - REKOMENDASI nohup java -jar target/procurement-approval-factory-1.0.jar --server.port=8081 > app.log 2>&1 & # nohup = no hang up, & = background
